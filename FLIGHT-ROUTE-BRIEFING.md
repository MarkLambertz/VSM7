## 1. What it is
A standalone, browser-native **process-flow editor** for VSM7 **Step VI (Channels)**, the **E2E Robustness Check**.
It is anchored to **one Success-Critical Task (SCT)** and lets the user lay out the **"flight route"** by which that SCT is
actually delivered across the organisation's **recursion levels** (Step I), in the spirit of Klaus Leopold's **Flight Levels**.

> **★ The SCT is the WHAT; the Flight Route is the HOW.**
> The SCT (and its per-recursion-level *contributions*) defines **what must be done**. The user builds the **flight route** —
> the activity flow that hops across recursion-level **swim-lanes** from a **user-defined Trigger** to a **Result** — i.e. **how** it is delivered.

Every lane-crossing is a recursion-level hand-off → a **VSM channel that must carry the work**. Making those crossings visible is the
point: a **human assesses** whether each hand-off / the whole route is robust. **The tool does not auto-score robustness** (see §7).

It is a **FULL EDITOR**: add / rename / move / delete steps, draw / delete connections, manage lanes, annotate with call-outs,
undo/redo, copy/paste, pan/zoom — the things a professional diagramming tool gives you.

## 2. Reference
The Bosch/"M-BD" **"Activity Cluster: Generic Example"** swim-lane slide Mark shared:
- Lanes top→bottom = recursion levels, most-aggregate at top → operational at bottom
  (`C` · `M/BD Overarching` · `M/BD Delivery` · `GB/BDO` · `Plant/ICO` · `Provider`).
- `Trigger of Work` (left) → chain of *"a step where something is done"* boxes → `RESULT` (right);
  arrows flow left→right and **hop between lanes** = the flight route (descend to operations, climb back to deliver).
- Two **call-out** kinds: (a) **context** on a step; (b) **highlight a transition between swim-lanes** ← the channel hand-off / robustness note.

## 3. The process — one, anchored to an SCT
- The check is built around **exactly one SCT** at a time (the WHAT).
- That SCT already carries **contributions** spread across recursion levels (Step III/IV). These are loaded as **context** — the
  building blocks of *what* happens at each level — available to drop into / reference while building the route.
- The **user defines the Trigger of Work** and assembles the **flight route** (the HOW): activity **steps** placed in lanes and
  connected into the end-to-end flow, ending at a **Result**. Steps are user-authored; contributions are the context/palette.

## 4. Objects (the model vocabulary)
- **Lane** — a recursion level (from Step I). Ordered, labelled, full-width band. **Editable here + synced back** (§8).
- **Step** — an activity. Snapped to a `(lane, column)` cell. Has a label; optionally linked to an SCT **contribution**.
- **Trigger / Result** — start / end of the route. Trigger label is **user-defined**.
- **Connection** — directed arrow (trigger→step, step→step, step→result). The ordered set = **the route**; cross-lane = **hand-off**.
- **Call-out** — annotation on a step or a connection; `kind = context | transition`. *Transition call-outs double as the human's robustness notes.*
- **Process** — exactly one, bound to the selected SCT.

## 5. It IS an editor — operations
**Canvas:** pan (drag empty bg) · zoom (wheel / `+`–`−` / fit) · **snap-to-grid** — rows = lanes, columns = sequence positions; steps snap to a cell. *(port pan/zoom + full-viewport canvas from `vsm.html`.)*
**Lanes:** **fully read-only** — **Step I owns the recursion-level set, its names AND its order**. Rendered here only as a reference list + the swim-lane bands. No add / remove / rename / reorder in this editor (all Step I). The one lane-related action is placing or dragging a *step* into a level, which assigns that step's recursion level. (The "+ Add level", per-lane delete, rename inputs and reorder ▲▼ were all removed 2026-06-19.)
**Steps:** add (double-click a cell / drag a contribution in) · inline rename (double-click) · properties panel · **drag to snap to another `(lane, column)`** — moving across lanes **re-assigns the recursion level** · delete · duplicate.
**Connections:** draw (drag from a step's handle to another step) · delete · re-route · optional label. Orthogonal/elbow routing, arrowheads, clear lane-crossings.
**Call-outs:** add to a step/connection · edit text · choose kind · delete.
**Selection & edit:** click · shift-multi-select · marquee · **undo/redo** (⌘Z / ⌘⇧Z) · copy/paste/duplicate · delete (⌫) · Esc cancels · arrow-key nudge.
**Persistence:** auto-save to `localStorage` (versioned key) · **export / import JSON** · export PNG/SVG (later).

## 6. Layout
- **Top toolbar (slim):** title · tools (select / add-step / connect / call-out) · actions (undo · redo · delete · fit · **▶ run flight**) · pane toggle.
- **Canvas:** swim-lanes with **lane labels down the left edge**; a light **column grid** (snap targets); the flow lives in the lane area; pan/zoom.
- **Right panel (collapsible — reuse the pane-collapse pattern):** *SCT* (the selected WHAT + its contributions as context) · *Recursion levels* (lane list + add/reorder) · *Properties* (selected object) · *Flight* (trigger/result, run/animate).

## 7. Simulation & human assessment
- **"Simulation" = a visual aid, not an auto-verdict.** Animate a token from Trigger along the links to Result (the "flight"), pulsing each lane-crossing, so the workshop can *see* the work descend to operations and climb back. Play / pause / step.
- **The assessment is the human's.** The tool surfaces the crossings (hand-offs) and may show **neutral facts** (e.g. # of crossings, lanes touched, orphan steps / unreachable Result) as input — but it does **not** label anything "robust/fragile". The user judges.
- The user **captures the assessment as call-outs** — especially **transition call-outs** marking a risky hand-off / missing channel. Those annotations ARE the robustness output of Step VI.

    // host side
    const pending = new Map();
    window.addEventListener('message', (e) => {
      if (e.source !== iframe.contentWindow) return;            // validate the source
      const m = e.data; if (!m || !m.evt) return;
      if (m.evt === 'exportReady') {
        const cb = pending.get(m.requestId); pending.delete(m.requestId);
        const url = URL.createObjectURL(m.blob);                // m.blob survives structured clone
        const a = document.createElement('a'); a.href = url; a.download = m.filename; a.click();
        URL.revokeObjectURL(url); cb && cb(null, m);
      } else if (m.evt === 'exportError') {
        const cb = pending.get(m.requestId); pending.delete(m.requestId); cb && cb(new Error(m.message));
      }
    });
    function exportRoute(format = 'png') {                       // returns a Promise<{blob,filename,mimeType}>
      const requestId = crypto.randomUUID();
      return new Promise((res, rej) => {
        pending.set(requestId, (err, m) => err ? rej(err) : res(m));
        iframe.contentWindow.postMessage({ cmd: 'export', format, requestId }, '*');
      });
    }
